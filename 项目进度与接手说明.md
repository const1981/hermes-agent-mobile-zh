## 一、一句话定位

做一个**能在安卓手机本地跑 Hermes Agent 调度层、仅大模型推理走云端 API** 的 App（产品化，点图标即用，不是 Termux 脚本）。当前基于 fork 的 `Binair-Dev/HermesAgentMobile`（Flutter 工程，MIT）做**中文版**。

## 三、技术架构（概览）

```
Flutter App (flutter_app/)
  ├─ UI 层 (lib/)：设置 Token / 渠道商 / 模型、分身开关、重启网关、一键安装
  ├─ 原生层 (android/ Kotlin)：前台服务 + MethodChannel（channelName=com.nxg.hermesagentmobile/native）
  └─ 调 Termux + proot-distro Ubuntu（首装约 300MB）
            │
            ▼
hermesx CLI (bin/ + lib/，Node)
  ├─ hermesx setup  → 克隆 Hermes 并 pip install -r requirements.txt
  └─ hermesx start  → 起 gateway/run.py（端口 18789）
            │
            ▼
Hermes Agent gateway（Python，跑在手机 Ubuntu 里，推理走云端 API）
```

- 网关主端口：`127.0.0.1:18789`（见 `flutter_app/lib/constants.dart`）
- 分身第二实例（规划中）：端口 `18790` + 独立 HOME + 独立人格 `SOUL.md`

## 四、关键决策记录（为什么这么选）

1. **中文版路线**：参考 Binair 思路，自己写中文代码，不直接抄；保留 MIT 合规。
2. **必须有 App**：臣哥硬性要求「点图标即用」，不能只是 Termux 脚本。
3. **Python 版本对齐**：Termux 的 pip 偏新，Hermes 支持的版本不同 → 解决＝始终用 Hermes 自带 `requirements.txt` 固定版本（`hermesx setup` 已这么做，不要手动升/降）。
4. **App 界面随时设置**：Token、渠道商、模型、分身开关、重启网关、一键安装，都做进原生 UI。
5. **分身实现 =「多人格分身实例（真功能）」**：开启时启第二个 Hermes 实例（18790 + 独立 HOME + 独立 `SOUL.md` 人格），关闭时只跑主实例。
   - ⚠️ 重要：Hermes **官方没有「分身/subagent 布尔开关」**，`delegate` 是模型自动能力。所以我们**不做假开关**，做真双实例。
7. **App 内一键安装（臣哥硬要求，2026-07-14）**：用户原话"最好是直接 app 一键安装，而不需要在软件 Termux 上面去用命令"。→ 安装由 App 内的「一键安装」按钮触发（App 代跑 `scripts/install-termux.sh`），**普通用户不打开 Termux、不手敲命令**。Termux 一条命令仅作为进阶 / 排障入口保留。所有文档与 UI 文案都要贯彻这一点。
8. **OpenRouter 无免费额度（2026-07-14）**：臣哥确认 OpenRouter 已无免费模型，原文档写的 `xiaomi/mimo-v2.5`（免费）**已从预设中删除**。预设改为 `openai/gpt-4o-mini` 并标注"需自备付费 key"。`README.md` / `config/config.yaml.example` / `scripts/install-termux.sh` 均已同步清理。
9. **config 结构已修正为官方 schema**（见 `config/config.yaml.example`）：
   ```yaml
   model:
     provider: ${PROVIDER}      # openrouter / zhipu / deepseek / kimi / custom
     model: ${MODEL}
     base_url: ${BASE_URL}
     api_key: ${API_KEY_REF}    # 引 .env 里的变量，不要明文写
   ```
   自定义 OpenAI 兼容端点设 `provider: custom`。**不要**写之前错误猜测的 `default:` / `api_mode: chat_completions`。

## 六、已知坑 / 接手必读（CI 构建环境）

> ⚠️ 本仓库的 GitHub Actions **实际跑在【本机 Windows 自托管 runner】上**，不是 GitHub 云 ubuntu。日志路径 `C:\Users\Administrator\AppData\Local\Android\Sdk`。

- 这台机器：装了 **Android Studio**（`D:\ProgramData\apk\bin\studio64.exe`），但**全机没有命令行版 `cmdline-tools` / `sdkmanager`**（Android Studio 的 SDK 管理只能走 GUI）。
- SDK 现状（`C:\Users\Administrator\AppData\Local\Android\Sdk`）：
  - `platforms/android-36`：**7/12 那次装崩留下的半残目录**，Gradle 认不出 → 想重装 → 撞陈旧 `.temp` → 报 `FileAlreadyExistsException`（ic_close.xml）。
  - `build-tools`：有 `36.1.0` / `37.0.0`（够用，无需 `36.0.0`）。
  - `ndk`：**未安装**；但 `flutter_app/android/app/build.gradle` 要求 `ndkVersion = "27.2.12479018"` → **必须装**。
  - `cmdline-tools/latest/bin`：**空**（无 sdkmanager）。
- **首次构建失败根因**：Gradle 自带 sdkmanager 装 `platforms;android-36` 时，撞上 SDK 陈旧 `.temp` 目录 → `FileAlreadyExistsException`。
- **修复方案（方案 A，本次采用，已验证 `dl.google.com` 可达）**：CI workflow 里
  1. 清 `.temp`；
  2. 下载 `commandlinetools-win-11076708_latest.zip` 解压到 `SDK\cmdline-tools\latest`；
  3. 删半残 `android-36` 重装；
  4. 装 `ndk;27.2.12479018`。
- **备选方案 B**（若你更想用 Android Studio GUI）：在 Android Studio 里装 NDK 27.2.12479018 + 修复 Platform 36，然后 workflow 只留「清 `.temp`」一步即可，不需要自动下载 cmdline-tools。

## 七、怎么本地构建 / 接手

- **CI 自动（推荐）**：改代码 → `git push` 到 `main` → GitHub Actions 在本机 runner 上构建 → 去 Actions 页面下载 Artifacts 里的 `app-release.apk`。
- **本机 Android Studio 直接构建**：用 Android Studio 打开 `flutter_app/android`，Sync 后 Build APK。前提：SDK 已装 `platform 36` + `ndk 27.2.12479018`。
- **改名约定**：App 显示名 = `Hermes Android App`（`AndroidManifest.xml` 的 `android:label` + `constants.dart` 的 `appName`）。App 界面需常驻显示版本号（`constants.dart` 的 `version`）。
