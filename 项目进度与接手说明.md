## 三、技术架构（概览）

```
Flutter App (flutter_app/)
  ├─ UI 层 (lib/)：设置 Token / 渠道商 / 模型、分身开关、重启网关、一键安装
  ├─ 原生层 (android/ Kotlin)：前台服务 + MethodChannel（channelName=com.nxg.hermesagentmobile/native）
  └─ 调 Termux + proot-distro Ubuntu（首装约 300MB）
            │
            ▼
hermesx CLI (bin/ + lib/，Node)
  ├─ hermesx setup  → 克隆 Hermes 并 pip install -r requirements.txt
  └─ hermesx start  → 起 gateway/run.py（端口 18789）
            │
            ▼
Hermes Agent gateway（Python，跑在手机 Ubuntu 里，推理走云端 API）
```

- 网关主端口：`127.0.0.1:18789`（见 `flutter_app/lib/constants.dart`）
- 分身第二实例（规划中）：端口 `18790` + 独立 HOME + 独立人格 `SOUL.md`

## 六、已知坑 / 接手必读（CI 构建环境）

> ⚠️ 本仓库的 GitHub Actions **跑在 GitHub 托管的 ubuntu-24.04 云端 runner 上**（标准云 runner，**不是**本机 Windows 自托管）。
> 判别依据（看构建日志）：`Image Release: ubuntu24/...`、`Working directory: /home/runner/work/...`、`shell: /usr/bin/bash`。
> 该 runner **没有预装 Android SDK**，构建前由 `build-apk.yml` 自行安装。

- **采用的方案（已写入 build-apk.yml，setup-android 步骤已验证成功）**：用官方 `android-actions/setup-android@v4` 装 cmdline-tools + 接受许可（它正确处理 classpath，避免手搓解压的坑），再显式装 `platform-tools` + `platforms;android-35` + `platforms;android-36` + `build-tools;36.0.0` + `ndk;27.2.12479018`。
- **env 必须用绝对路径**：`ANDROID_SDK_ROOT=/home/runner/Android/Sdk`。**不要写 `$HOME/Android/Sdk`**——在 job env 里 `$HOME` 不会被展开，会变成字面量 `$HOME`（已踩过隐患）。
- **历史上踩过的坑（留作警示）**：
  - 误以为跑在本机 Windows 自托管 runner → 错设 `ANDROID_SDK_ROOT=C:/Users/...` + 下 `commandlinetools-win` + `cmd //c sdkmanager.bat` → Linux 上 `cmd: command not found`（exit 127）。
  - 曾用 `shell: powershell` 写修复步骤 → runner 是 bash，报 `powershell: command not found`。
  - 手搓解压 `commandlinetools-linux` 并调 `sdkmanager` → `Could not find or load main class com.android.sdklib.tool.sdkmanager.SdkManagerCli`（classpath jar 在云端 runner 上解析异常；zip 本身没问题，是手搓解压/调用不可靠）→ 改用 `android-actions/setup-android@v4` 解决（已验证该步成功）。
  - **`yes | sdkmanager --licenses` 的 SIGPIPE 坑（第 6 次构建 eb185e0 的失败点）**：许可接受完后 `yes` 继续写管道被 SIGPIPE 打断，退出码 1 → 配合 `set -e`，脚本在真正的 `sdkmanager "platform-tools"...` 安装命令之前就中断。修法：`yes | sdkmanager --licenses || true`（许可其实已被 setup-android 默认接受，这行只是兜底）。
  - **App 源码用了 Flutter 3.27+ 的 API（第 6 次构建 eb185e0 暴露的真问题）**：`flutter_app/lib/app.dart` 用了 `CardThemeData` / `DialogThemeData`（Flutter 3.27+ 才有），但 CI 固定 Flutter **3.24.5**（3.24 里叫 `CardTheme` / `DialogTheme`）。编译报错 `Error: The method 'CardThemeData' isn't defined for the class 'HermesAgentApp'`。修法：把这 4 处改为 `CardTheme` / `DialogTheme`（已修，提交 `1f4e21a`）。
- `flutter_app/android/app/build.gradle` 要求 `ndkVersion = "27.2.12479018"`，必须装对应 NDK（否则 Gradle 报 NDK not found）。`flutter_app/android/build.gradle` 的 `subprojects` 给插件设 `compileSdkVersion=35` → 还要装 `platforms;android-35`（已补）。
- **CI 当前状态（2026-07-14 14:22）**：前 6 次构建失败（5 次 SDK 安装问题 + 1 次 app.dart API 版本问题），已全部修复；第 7 次构建（commit `1f4e21a`，run `29311147258`）正在云端 runner 运行，待转绿。

## 十一、用户实测报错与修复记录（2026-07-14，真机反馈）

> 以下均为臣哥在不同手机上实测 APK 遇到的真实报错，已逐一读代码定位根因。

### 问题 1：dpkg was interrupted（旧版 0.2.4 实测）
- **现象**：proot 里 `dpkg --configure -a` 提示需手动运行；安装中断卡死。
- **根因**：apt-get 中途断网/被杀后台，dpkg 状态库锁死。
- **修复**：`bootstrap_service.dart` apt-get 前插 `dpkg --configure -a || true`（已合入 0.3.0，commit `5fc4224`）。
- **状态**：✅ 已修复，待新版 APK 验证。

### 问题 2：pip install --upgrade pip 报 PROOT_ERROR exit code 1（新版手机实测）
- **现象**：`PlatformException(PROOT_ERROR, Command failed (exit code 1): Requirement already satisfied: pip ... Collecting pip)`
- **根因**：`bootstrap_service.dart` 第 197 行 `pip install --upgrade pip && pip install -r requirements.txt` 用 `&&` 链。proot 里 `--upgrade pip` 返回非零退出码（即使已装好）→ 后续装依赖命令被跳过 → 整条安装链断裂。
- **修复**：`&&` → `|| true &&`（commit `232298a` 已推送）。
- **状态**：✅ 已修复，待 CI 出新 APK 验证。

### 问题 3：桌面安卓系统（x86_64）打开闪退
- **现象**：臣哥在桌面安卓电脑（x86_64 架构）装 APP 即闪退，打不开。
- **根因（推测）**：`runInProot` 内置的 proot 二进制是 **arm 版**，x86_64 桌面安卓无法执行；或 rootfs 仅 arm 兼容。
- **状态**：🔴 未修。需查 `android/app/src/main/` 下 proot 资源打包（是否仅 armeabi-v7a/arm64-v8a）并补 x86_64 版，或明确声明不支持 x86。

### 问题 4：第二次打开仍重装 Ubuntu rootfs
- **现象**：已跑过一遍安装，下次打开又重新下载/解压 rootfs，不从断点续跑。
- **根因（读代码确诊）**：`runFullSetup()` 无幂等守卫，无条件重下 rootfs；`extractRootfs`（Kotlin `BootstrapManager.kt`）开头 `deleteRecursively(rootfs)` 删已装环境；`isBootstrapComplete()` 读文件系统 4 文件而非 sp 标记。
- **理想修复（臣哥要求"下次直接继续"）**：A. `runFullSetup` 开头判 `isBootstrapComplete` 跳过；B. 步骤级断点续传（rootfs 已解包跳下载/解包，python3 已在跳 apt，hermes 已 clone 跳 git，venv 已建跳 pip）；C. `extractRootfs` 已有完整 rootfs 则 return 不删。
